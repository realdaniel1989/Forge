#01 Unit Conversion Destroys Weight Precision
Resolved LiveWorkout.tsx Data Integrity
✓ Fixed — 2026-05-18

All weights are now stored internally in a fixed base unit (lbs). Conversion only happens at the display boundary — toggling the unit selector no longer mutates stored weight data.

// Added: base unit ref + conversion helpers (display-only) const baseUnitRef = useRef<'lbs' | 'kgs'>(unit); const toDisplayUnit = (weight) => { if (baseUnitRef.current === unit) return weight; return unit === 'kgs' ? weight / 2.20462 : weight * 2.20462; }; const fromDisplayUnit = (weight) => { /* inverse */ }; // handleUnitChange now only changes the display unit: const handleUnitChange = (newUnit) => { if (newUnit === unit) return; setUnit(newUnit); // no weight mutation }; // Weight input uses conversion at boundary: value={parseFloat(toDisplayUnit(set.weight).toFixed(1))} onChange={e => updateSet(..., fromDisplayUnit(parseFloat(...)))} // Round-trip verification: 135 lbs → kgs → lbs = 135.0 (zero loss)

Original Problem

The handleUnitChange function converted all weight values in-place using Math.round(). Because rounding is lossy, converting back and forth accumulated errors.

// BEFORE — mutated stored data on toggle weight: newUnit === 'kgs' ? Math.round(set.weight / 2.20462) : Math.round(set.weight * 2.20462) // Data loss chain: 135 lbs → 61 kg → 134 lbs (lost 1 lb)
#02 Abort Workout = Silent Data Loss
Resolved LiveWorkout.tsx App.tsx Data Loss
✓ Fixed — 2026-05-18

Three-part fix applied:

1. Auto-save to localStorage — Every state change persists the full workout to localStorage under a key scoped to the routine. Protects against browser refresh, accidental back-navigation, and app crashes.

2. Draft resume on mount — On component mount, checks for an existing draft and restores exercises, unit, and base unit. Users pick up exactly where they left off.

3. Abort confirmation modal — Replaced the direct onFinish() call with a custom confirmation modal styled to match the app's dark theme. Works in iframes (unlike window.confirm). "Continue Workout" dismisses; "Discard" clears the draft and exits.

// Auto-save on every state change useEffect(() => { localStorage.setItem(draftKey, JSON.stringify( { exercises, unit, baseUnit: baseUnitRef.current } )); }, [exercises, unit]); // Resume from draft on mount const savedDraft = (() => { const raw = localStorage.getItem(draftKey); return raw ? JSON.parse(raw) : null; })(); const [exercises, setExercises] = useState( savedDraft?.exercises || routine.exercises.map(...) ); // Clear draft on successful save localStorage.removeItem(draftKey); onFinish();

Original Problem

Clicking "Abort" called onFinish() directly, immediately unmounting the component with no confirmation. Workout state existed only in React component state — gone on unmount, browser refresh, or navigation.

// BEFORE — no protection <button onClick={onFinish}>Abort</button> // ↑ instant unmount, all data destroyed
#03 Firestore Save Failure Silently Loses Completed Workout
Resolved LiveWorkout.tsx firestoreUtils.ts Data Loss
✓ Fixed — 2026-05-19

Three-part fix applied:

1. Error state with retry UI — Added a saveError state. When finishWorkout() fails, the error is caught and displayed in a fixed banner at the top of the screen with a "Retry" button. The workout data stays in component state so the user can retry saving.

2. No more re-thrown unhandled errors — Removed the handleFirestoreError() call from finishWorkout() (which re-threw and became an unhandled promise rejection). Replaced with console.error() and user-facing error message.

3. Data survives failures — The localStorage auto-save draft (from Issue #2 fix) ensures workout data is backed up locally. The user can retry or refresh without losing their workout.

// finishWorkout now catches errors gracefully const [saveError, setSaveError] = useState<string | null>(null); const finishWorkout = async () => { setSaving(true); setSaveError(null); try { await addDoc(...); localStorage.removeItem(draftKey); onFinish(); } catch(e) { console.error('Failed to save workout:', e); setSaveError('Failed to save. Your data is safe — tap Retry.'); } finally { setSaving(false); } };

Original Problem

When finishWorkout() failed (network error, permission denied, etc.), handleFirestoreError() re-threw the error but nothing caught it — becoming an unhandled promise rejection. The user saw nothing and the workout data was lost.

#08 NaN in Footer Stats for Cardio Exercises

✓ Fixed — 2026-05-19

The footer stats now use pre-computed completedSets and totalSets variables that handle cardio exercises correctly. Cardio exercises are excluded from total sets count, and the completed count uses || 0 fallback for exercises without trackedSets.

// Pre-computed in component body const completedSets = exercises.reduce((acc, ex) => acc + (ex.trackedSets?.filter(s => s.completed).length || 0), 0); const totalSets = exercises.reduce((acc, ex) => acc + (ex.type === 'cardio' ? 0 : (ex.trackedSets?.length || ex.sets || 0)), 0);

#11 Expensive History Fetch in LiveWorkout (O(n×m))
Resolved LiveWorkout.tsx Performance
✓ Fixed — 2026-05-19

Added limit(10) to the history fetch query. Since logs are sorted by date descending, the first match per exercise is still the most recent usage. This reduces the iteration from O(n × m) where n = total lifetime logs to O(10 × m).

// Before: fetched ALL logs ever const q = query(collection(db, 'workoutLogs'), where('userId', '==', user.uid)); // After: only 10 most recent logs const q = query(collection(db, 'workoutLogs'), where('userId', '==', user.uid), limit(10));

Original Problem

LiveWorkout fetched all workout logs on mount and iterated every exercise in every log to build a history lookup map.

#12 Full Array Copy on Every Keystroke in Exercise Updates
LiveWorkout.tsx Performance

Problem

Every call to updateSet, addSet, deleteSet, etc. creates a full copy of the entire exercises array. For weight/reps inputs, this fires on every keystroke, creating many intermediate arrays that are immediately garbage collected.

// LiveWorkout.tsx — updateSet() const updateSet = (exerciseIndex, setIndex, field, value) => { const newEx = [...exercises]; // copy full array const newSets = [...(newEx[exerciseIndex].trackedSets || [])]; // copy sets newSets[setIndex] = { ...newSets[setIndex], [field]: value }; newEx[exerciseIndex] = { ...newEx[exerciseIndex], trackedSets: newSets }; setExercises(newEx); };
Recommended Fix

For input-heavy fields (weight, reps), debounce the state updates so the array isn't copied on every keystroke. Alternatively, use useReducer for more granular state management that can update a single exercise without copying the whole array.
