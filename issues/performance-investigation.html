Performance Investigation

Forge.AI Gym Workout Tracker · 2026-05-20

Root Cause Summary

The perceived lag on every click/tab-switch is caused by components fully unmounting and remounting on every tab change. Each remount triggers a fresh Firestore network query with a loading spinner. There is no data caching, no state persistence, and no real-time listeners. Every interaction is a cold start.

Critical Components Fully Remount on Every Tab Switch

src/App.tsx → MainView → lines 15–22

// Current: conditional rendering = unmount when hidden {tab === 'routines' && <RoutinesList ... />} {tab === 'generate' && <RoutineGenerator ... />} {tab === 'custom' && <CustomRoutineBuilder ... />} {tab === 'progress' && <ProgressView />}

When tab changes from 'routines' to 'progress', React unmounts RoutinesList completely (DOM destroyed, state discarded) and mounts ProgressView from scratch. Going back to routines means another full mount → useEffect → Firestore query → loading spinner → render.

User-perceived lag chain: Click tab → component unmounts → new component mounts → useEffect fires → setLoading(true) → loading spinner renders → Firestore network request (100–800ms) → data arrives → setLoading(false) → content renders. Total: 200ms–2s of visible delay per click.
This is the primary cause of the sluggish feel. Every single tab switch is a cold start with a network round-trip.

Recommended Fix

Option A — Keep components mounted, hide with CSS:

// Wrap each tab in a div, toggle visibility with CSS <div style={{ display: tab === 'routines' ? 'block' : 'none' }}> <RoutinesList ... /> </div> <div style={{ display: tab === 'progress' ? 'block' : 'none' }}> <ProgressView /> </div>

Components stay mounted (state preserved, no re-fetch). Only the active tab is visible. First load still fetches data, but subsequent tab switches are instant.

  • Option B — Lift data fetching to App level and pass data down as props
  • Option C — Use a global state store (React Context, Zustand) to cache Firestore data

Critical No Firestore Data Caching or Real-Time Listeners

src/components/RoutinesList.tsx, src/components/ProgressView.tsx, src/components/LiveWorkout.tsx

Every data-fetching component uses getDocs() — a one-shot fetch with no caching. Firestore SDK has built-in offline persistence and onSnapshot() real-time listeners, but neither is used.

// RoutinesList.tsx — fetches 50 routines on every mount const q = query(collection(db, 'routines'), where('userId', '==', user.uid), limit(50)); const snap = await getDocs(q); // ← always hits network // ProgressView.tsx — fetches 100 logs on every mount const q = query(collection(db, 'workoutLogs'), where('userId', '==', user.uid), limit(100)); const snap = await getDocs(q); // ← always hits network // LiveWorkout.tsx — fetches 10 logs on every mount const q = query(collection(db, 'workoutLogs'), where('userId', '==', user.uid), limit(10)); const snap = await getDocs(querySnapshot); // ← always hits network
No offline support. Every visit to a tab is a guaranteed network round-trip. Even if the user just visited that tab 2 seconds ago, all data is discarded and re-fetched. On slow connections (3G, weak WiFi), this means multi-second freezes.

Recommended Fix

  • Enable Firestore offline persistence: initializeFirestore(app, { localCache: persistentLocalCache() })
  • Switch to onSnapshot() real-time listeners for lists that need to stay fresh — data arrives instantly from cache, updates in real-time from server
  • At minimum, lift fetched data up to a shared context so it survives tab switches

High ProgressView Verbose Debug Logging

src/components/ProgressView.tsx → lines 33–47

// This runs on every mount (every tab switch to Analytics) console.log('[Analytics] Fetched logs:', fetched.length); fetched.forEach((log, i) => { console.log(`[Analytics] Log ${i}:`, { name: log.name, date: log.date, dateStr: new Date(log.date).toISOString(), bodyPart: log.bodyPart, exerciseCount: log.exercises?.length, exercises: log.exercises?.map(ex => ({ name: ex.name, type: ex.type, trackedSets: ex.trackedSets?.length, completedSets: ex.trackedSets?.filter(s => s.completed).length, actualSets: ex.actualSets, sets: ex.sets, reps: ex.reps, trackedSetDetails: ex.trackedSets, // ← dumps entire tracked sets })) }); });

For 100 workout logs with ~6 exercises each, this creates ~600+ console.log calls on every Analytics tab visit. Each log serializes full exercise data including tracked sets. On mobile devices, console.log is synchronous and blocks the main thread.

Impact: 100 logs × 6 exercises = 600+ console writes. On Chrome DevTools open, this causes visible jank. On mobile Safari, it can cause 100–300ms of main-thread blocking.

Recommended Fix

Remove all debug console.log calls from ProgressView.tsx. They are development artifacts that should not be in production.

High LiveWorkout localStorage Sync on Every State Change

src/components/LiveWorkout.tsx → lines 60–63

// Fires on EVERY exercises or unit change useEffect(() => { try { localStorage.setItem(draftKey, JSON.stringify({ exercises, unit, baseUnit: baseUnitRef.current })); } catch {} }, [exercises, unit]);

The exercises array contains the full workout state — every exercise with every tracked set. When a user types in a weight or reps input, updateSet() creates a new exercises array → this effect fires → JSON.stringify serializes the entire workout state → writes to localStorage. This is synchronous and runs on every keystroke.

For a workout with 8 exercises × 4 sets each, that's serializing 32+ tracked sets on every single number input change.

Impact: JSON.stringify of complex objects on every keystroke. Causes input lag especially on mobile devices where localStorage writes are slower. Combined with React re-renders, each number change triggers: state update → re-render → effect → serialize → write.

Recommended Fix

  • Debounce the save: Use a 500ms debounce so rapid inputs don't each trigger a save
  • Use useRef for the draft and only serialize on unmount or periodically
// Debounced version useEffect(() => { const timer = setTimeout(() => { try { localStorage.setItem(draftKey, JSON.stringify({ exercises, unit, baseUnit: baseUnitRef.current })); } catch {} }, 500); // ← only saves after 500ms of inactivity return () => clearTimeout(timer); }, [exercises, unit]);

Medium AuthContext Test Connection Query

src/AuthContext.tsx → lines 15–20, 30

// Fires on every AuthProvider mount (= every app load) async function testConnection() { try { await getDocFromServer(doc(db, 'test', 'connection')); // ← Firestore read } catch (error) { if(error instanceof Error && error.message.includes('the client is offline')) { console.error("Please check your Firebase configuration."); } } } // ... useEffect(() => { testConnection(); // ← called immediately on mount const unsubscribe = onAuthStateChanged(auth, (currentUser) => { ... }); return () => unsubscribe(); }, []);

Every app load fires a Firestore read to a 'test/connection' document that likely doesn't exist. This adds latency to the initial auth check. The getDocFromServer call explicitly bypasses cache and hits the network.

Recommended Fix

Remove the testConnection() call entirely. Firebase SDK already handles connectivity errors gracefully — the onAuthStateChanged listener and Firestore operations will fail with clear error messages if there's no connection. There's no need for a separate connectivity probe.

Low Array Index as React Key

src/components/LiveWorkout.tsx, src/components/ProgressView.tsx

// LiveWorkout.tsx — exercises rendered with index key {exercises.map((ex, idx) => { return ( <div key={idx} ...> // ProgressView.tsx — weekly summary with index key {Object.entries(weeklySummary).map(([part, val], i) => ( <tr key={i} ...>

Using array index as key prevents React from efficiently reusing DOM nodes when list items are reordered, added, or removed. In LiveWorkout, exercises can be reordered (move up/down) and deleted — index keys cause React to re-render more DOM than necessary.

Recommended Fix

  • For exercises in LiveWorkout: use ex.name + idx or a generated stable ID
  • For ProgressView summary rows: use the part string as key (it's already unique)

Architecture Recommendation

The root architectural issue is that data fetching is tightly coupled to component lifecycle. Each component owns its own data, fetches on mount, and loses it on unmount. This pattern is fine for one-time views, but for a tabbed app where users switch frequently, it creates a terrible UX.

The ideal fix introduces a data layer that:

  • Fetches data once and caches it globally (not per-component)
  • Uses Firestore onSnapshot() listeners to keep data fresh
  • Keeps tab components mounted but visually hidden
  • Provides optimistic UI for writes (don't wait for server on delete/save)
