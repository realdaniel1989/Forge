Bloat Audit — gym-workout-tracker

Generated 2026-05-21 · React 19 + Vite + Firebase + Express + Gemini · 3,938 LOC source

TL;DR

The app itself isn't huge — ~3.9K LOC across 14 components. The bloat lives in three places:

  1. Firebase v12 dominates the 855 KB client bundle (~60–70% of it). It's also the biggest folder in node_modules at 148 MB.
  2. Ghost dependencies in package.json: recharts, motion, and date-fns are partially or fully unused in source. @google/genai pulls 8+ MB of transitive deps (protobufjs, web-streams-polyfill, framer-motion) into node_modules.
  3. No code splitting. One 855 KB JS chunk on first load. Two screens (LiveWorkout 809 LOC, ProgressView 747 LOC) are eagerly loaded even before login.

Direct answer to your question: migrating Firebase → Supabase is a real win (~250–400 KB → ~80–120 KB on the bundle), but it's not the first thing to do. You can cut ~250 KB and ~400 MB of node_modules in an afternoon without touching the database. Refactoring the whole app is not required.

The bloat issues, ranked by ROI

#1 — Ghost dependencies 15 min · saves ~30–60 KB bundle, ~25 MB node_modules

recharts and motion appear in package.json but have zero imports anywhere in src/ or server.ts. Tree-shaking probably keeps them out of the final JS, but they still bloat installs, lockfiles, and CI cache.

Fix:

npm uninstall recharts motion

If you intended to use Recharts for ProgressView, decide now: add it back deliberately, or stick with the current hand-rolled chart code.

#2 — No code splitting 1–2 hr · saves ~400–500 KB from initial load

Every screen is statically imported in App.tsx. That means a logged-out user downloads LiveWorkout (809 LOC), ProgressView (747 LOC), and ExerciseLibrary (425 LOC) before they see the login button.

Fix: lazy-load route-level components.

// src/App.tsx
import { lazy, Suspense } from 'react';
const LiveWorkout    = lazy(() => import('./components/LiveWorkout'));
const ProgressView   = lazy(() => import('./components/ProgressView'));
const ExerciseLibrary= lazy(() => import('./components/ExerciseLibrary'));
const HabitTracker   = lazy(() => import('./components/HabitTracker'));

// Wrap the router/switch:
<Suspense fallback={<div className="p-8">Loading…</div>}>
  {/* routes */}
</Suspense>

Also add manual chunking for Firebase in vite.config.ts:

build: {
  rollupOptions: {
    output: {
      manualChunks: {
        firebase: ['firebase/app','firebase/auth','firebase/firestore'],
        react:    ['react','react-dom'],
      },
    },
  },
}

Expected: initial JS drops from 855 KB → ~300–350 KB; Firebase chunk loads in parallel and is cached separately across deploys.

#3 — Firebase SDK is the heaviest piece left decision point

Even with chunking, Firebase Auth + Firestore is roughly 250–400 KB gzipped at runtime. It's the biggest single thing on the wire.

Option A — Stay on Firebase, trim it. You're using only Auth and Firestore. Keep imports modular (you already do this — good). Don't pull in firebase/analytics, firebase/storage, etc. No further code change needed beyond chunking.

Option B — Move to Supabase. @supabase/supabase-js is ~80–120 KB gzipped. Net bundle savings: ~150–250 KB. Migration cost:

  • Schema: routines, workoutLogs, exerciseLibrary, habits become Postgres tables. firestore.rules (168 lines) becomes Row-Level Security policies — roughly 1:1 conceptually.
  • Auth: Supabase Auth has Google OAuth out of the box; AuthContext rewrite is ~50 lines.
  • Realtime: you don't appear to use onSnapshot heavily; getDocs dominates. So Postgres+REST is fine.
  • Offline: you use persistentLocalCache. Supabase doesn't ship equivalent offline persistence — you'd add IndexedDB caching yourself if it matters.

Option C — Postgres directly on Railway. Smallest possible client (just postgres driver or REST), but you'd build auth, RLS, and the API layer yourself. ~1–2 weeks of work for marginal gains over Supabase.

My recommendation: do Options #1, #2, #4 first. Then measure. If your bundle is already ~300 KB and Firestore reads are fast, Supabase is not worth the migration. The win there is mostly cost predictability + portability, not raw performance.

#4 — date-fns is overkill for 7 functions 20 min · saves ~10–20 KB bundle, 42 MB node_modules

Two files import 7 named functions: format, subDays, eachDayOfInterval, startOfWeek, endOfWeek, subWeeks, addWeeks, isSameDay.

Options:

  • Replace with dayjs (~7 KB total): drop-in for format, plus weekday/isSame plugins.
  • Write 30 lines of native JS. Intl.DateTimeFormat + a tiny src/dateUtils.ts covers all 7 use cases. Zero deps.

Vite already tree-shakes date-fns at the bundle layer, but the 42 MB node_modules footprint slows installs and CI. Replacing it is mostly a hygiene win.

#5 — @google/genai is pulling in junk 30 min · saves 25–40 MB node_modules

The Gemini SDK is used only in server.ts (5 lines). It drags in protobufjs, web-streams-polyfill, framer-motion(!), @reduxjs, es-toolkit — none of which show up in your source.

Since you only call ai.models.generateContent with a JSON schema, you can replace the SDK with a direct REST call to https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent using fetch. ~40 lines.

This doesn't shrink the client bundle (SDK never reaches the browser), but it removes ~40 MB of transitive cruft from node_modules and speeds up Railway's Nixpacks build (smaller install, less to cache).

#6 — LiveWorkout.tsx (809) and ProgressView.tsx (747) are oversized refactor, not bloat

Not strictly "bloat" — but these files are doing too much. Likely candidates for extraction:

  • LiveWorkout: split timer/rest logic, set tracking, and workout completion into hooks (useWorkoutTimer, useSetTracking).
  • ProgressView: split chart rendering from data aggregation.

Smaller files are also easier to lazy-load and tree-shake.

#7 — Misc cleanup trivial

  • src/.DS_Store is committed. Add **/.DS_Store to .gitignore.
  • firebase-applet-config.json, firebase-blueprint.json, metadata.json — left over from AI Studio scaffolding. Confirm they're not used and delete.
  • package.json has duplicate vite in both dependencies and devDependencies. Remove from dependencies.
  • @vitejs/plugin-react and @tailwindcss/vite live under dependencies — should be devDependencies. (Build-time only.)

Recommended order of operations

  1. Day 1 (under an hour): remove ghost deps (#1), fix package.json layout (#7).
    Result: cleaner installs, no behavior change.
  2. Day 1 afternoon (1–2 hours): add code splitting + manual chunks (#2).
    Result: first-paint JS drops from 855 KB → ~300–350 KB.
  3. Day 2: replace date-fns (#4) and inline the Gemini REST call (#5).
    Result: 60+ MB off node_modules, smaller Docker image.
  4. Measure. Run Lighthouse and check actual TTI. If you're under 2s on 4G, stop here.
  5. Only if needed: evaluate Supabase migration (#3, Option B). At that point you'll have hard numbers to justify the week of work.
  6. Background: refactor LiveWorkout and ProgressView when you next touch them (#6).

What you should not do

  • Don't rewrite the app. 3,938 LOC across 14 components is fine. The architecture is reasonable.
  • Don't migrate to Supabase as the first step. You'll spend a week and your bundle will still be 500 KB because you skipped code splitting. Fix the cheap stuff first; the expensive stuff may become unnecessary.
  • Don't drop Firebase Auth for a custom solution. You'll save bundle bytes and lose weeks. Auth is not where the wins are.
  • Don't move to Postgres-on-Railway directly unless you have a strong product reason (cost, data ownership, complex relational queries). For this workload, the perf delta vs. Firestore is invisible.
